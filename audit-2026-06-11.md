# Part 2 — New findings (commits `5a31765..819ec1d`)

## Bugs / correctness

### N1. Main-thread `convert()` can permanently lock the pipeline — HIGH
`features/image/controller.ts:457-512`. `convert()` sets `processing = true` and has **no try/catch**. If `renderImageBaseRaster` (pica), `getImageData`, or `new SharedArrayBuffer` throws, the rejection is swallowed by `void convert()` in the rAF and `processing` stays `true` forever — every later slider tweak only sets `processRequested`. This is exactly the failure class commit `f6ce58e` fixed for the *worker* side (code-map §8: "Without this contract a worker throw would lock the convert pipeline"); the symmetric main-thread path was missed.

**Fix:** wrap the body after `processing = true` in try/catch — on throw: `processing = false`, `host.showError(...)`, re-pump if `processRequested`. ~10 lines, mirrors the existing `worker.onError` handler.

### N2. Histogram zones and threshold markers ignore the quant preset — MEDIUM
The hidden Ctrl+Shift+Q toggle (`819ec1d`) switches dither quantization between PR1614 thresholds `[42,127,212]` and master `[45,70,140]` — but:
- `domain/histogram.ts:13-18` — `ZONE_TEMPLATE` hardcodes the PR1614 boundaries, so the four zone percentages are computed against the wrong bins under `master`.
- `infra/canvas/histogramRenderer.ts:86` and `:115` — marker lines and labels hardcode `[42, 127, 212]`.

Under the `master` preset the histogram panel reports zone percentages and boundaries that don't match what the dither actually does.

**Fix:** make `computeHistogramZones` and `renderHistogram` take the threshold triple (export it from `domain/quantize.ts` per preset), thread it from the worker result path the same way `quantPreset` already rides `WorkerSettings`.

### N3. The persisted quant preset is invisible — MEDIUM (UX trap)
`quantPresetToggle.ts` persists the preset to localStorage. The only feedback is a 1.6 s toast at toggle time. A user (or the author, weeks later) who once pressed Ctrl+Shift+Q is silently on `master` thresholds in every future session with zero indication. **Fix:** render a small persistent badge (footer or histogram corner) whenever the active preset ≠ default. One render hook + one element.

### N4. `domain/quantize.ts` holds module-level mutable state — MEDIUM (architectural)
`activePreset` is a hidden global inside the domain layer. It works today only because two separate module instances exist (main bundle + worker bundle) and the worker re-sets it on every `process` message. But: `quantize()`'s result now depends on invisible state; the vitest suite covers only the default preset and any future test calling `setQuantPreset` leaks state across tests; and domain "pure TS" purity (§2) is broken in spirit. **Fix:** pass the preset (or the threshold triple) as a parameter — `quantize(v, thresholds)` or a `makeQuantizer(preset)` — and delete `set/getQuantPreset`. Callers: `dither.ts` (worker) and the protocol already carries the preset, so the threading is mechanical.

### N5. Dead actions that are also the documented trap — LOW
`actions.ts:52/:54` + `reducer.ts:12/:16`: `imageSetMode` and `imageSetFitLockNative` are defined but never dispatched — they're precisely the two-step dispatch that §12 bans ("Don't dispatch mode + fitLockNative as two separate actions"). Leaving them exported invites the exact bug the atomic `imageSetModePreset` exists to prevent. **Fix:** delete both actions and reducer cases.

### N6. Max-fit-percent math duplicated across `ui/` — LOW
The same computation — rotated-dims swap → `min(targetW/sW, targetH/sH)` → `max(10, floor(100/maxFit))` — appears in `ui/render.ts:66-72` and `ui/bindings.ts:50-64` (and the rotated-swap fragment a third time in `ui/imageCropBridge.ts:getRotatedSourceDims`). §3 single-source violation + §12 "don't compute scales inline". **Fix:** `domain/geometry.ts:computeMaxFitSizePct({sourceW, sourceH, rotation, targetW, targetH})` + a `rotatedDims()` helper; add unit tests.

## Contract vs. enforcement divergence

### N7. depcruise encodes a looser rule than code-map §2
`.dependency-cruiser.cjs:25-35` exempts the wiring trio (`bootstrap`, `appController`, `loaderRouter`) from **both** `features/` and `ui/` imports. Code-map §2's wiring exception only blesses "feature factories/types". So `appController → ui/dom` (old finding 2) passes the build while violating the written contract. Pick one:
- (a) Implement old finding 2 (two canvas refs instead of `AppDom`) and narrow the depcruise exemption for `appController`/`loaderRouter` to `features/` only; or
- (b) Amend §2 to say the trio may also import ui types, and close old finding 2 as won't-fix.
(a) is ~15 lines and keeps the stricter invariant; recommended.

### N8. The §2 ui-litmus now contradicts §12
§2: "ui — am I about to read `store.getState()` directly? Stop; take a getter via deps." §12 (newer): no-op guards "live at the click-handler site in `ui/bindings.ts`" and are written as `store.getState()` comparisons. Both can't hold. The codebase has clearly voted: 19 direct reads across three ui modules, all store-via-deps. **Fix (doc):** rewrite the litmus to the rule actually being followed — "ui receives the store via deps; reads for guards/render are fine; never compute geometry or dispatch derived state from raw reads." Then old finding 9 closes as won't-fix.

### N9. Mirror toggles write the box outside the blessed writers — LOW
`controller.ts:181/:189` (see Part 1, finding 3). Either reroute through an `applyGeometry` box-override (old plan, step 3) or — cheaper — amend §5 to bless the mirror reposition explicitly. The write is sound (it feeds `applyGeometry`'s prev-center math on the very next call); this is a doc/contract decision, not a bug.

# Resolution log — 2026-06-11

**Fixed (bugs):** N1 — `convert()` wrapped in session-guarded try/catch mirroring `worker.onError`; the single-flight gate can no longer stick. N2 — histogram zones, marker lines, and labels take the quant-threshold triple (`getQuantThresholds(state.quantPreset)`); `ZONE_TEMPLATE` hardcoding removed. Tone-LUT gamma no longer rounds the black/white-remap output to an integer.

**Fixed (conformance/cleanup):** Old findings 1, 2, 4 implemented as planned (`clearHistogram` is canvas-only with `unloadImage` owning `lastHistogram`; `appController` takes two canvas refs instead of `AppDom`; `clearOutputBytes` call dropped from app). N4 — `domain/quantize.ts` is pure: thresholds are parameters, the active preset lives in `state.quantPreset`, the worker resolves them from `WorkerSettings.quantPreset`. N5 — dead `imageSetMode`/`imageSetFitLockNative` actions deleted. N6 — max-fit math deduped into `domain/geometry.ts:computeMaxFitSizePct` + `rotatedSourceDims`. N7 — depcruise gained `app-wiring-no-ui` (ui imports are bootstrap-only). N3 — persistent `.quant-badge` renders whenever the persisted preset ≠ default.

**Resolved doc-side:** N8 — §2 ui-litmus rewritten (deps-provided store reads are fine; derived business math is not), closing old finding 9 as won't-fix. N9 — mirror's `setBoxPosition` blessed in §5. G6 — code-map synced (§1 BMP depth, §8/§9 `sessionVersion`, §6/§10 quant + geometry entries, appendix build line).

**New fix found during implementation:** `loaderRouter` now unloads the opposite feature's live session before routing (and pasted printer text routes through the router). Previously, loading a GB file over a live image session left the image worker in flight; its late result could repaint the preview over the freshly built GB output.

**Still open (enhancements, not approved):** E1–E7, CI workflow, `.gitignore` hygiene.

## Round 2 — 2026-06-11 (same day, later pass)

Second audit pass over the previously-unread modules (GB decoders, BMP encoders, infra adapters, small ui binders, full index.html) plus a device-switch/async-failure regression trace. Encoder and pixel math all verified correct. Found and fixed:

- **F1 (MEDIUM)** — GB `outputScale` was never re-clamped on device change; `buildGbOutputArtifacts` silently cropped the art (e.g. a 160×400 printer strip at 2× lost 8 px switching X4→X3). `buildOutput` now clamps via the new `domain/gb/displayScale.ts:computeMaxGbOutputScale` and dispatches the corrected scale; `adjustOutputScale` reuses the helper (and no longer rotates the full pixel buffer just to read dims).
- **F2 (LOW-MEDIUM)** — device switch mid-convert left `lastIndexedPixels` sized for the old device: the worker `onResult` painted a corrupt preview frame and flagged outputs ready (F2a), and `downloadActive` could encode a structurally corrupt file during the window (F2b). Both sites now size-guard against `device.totalPixels`; the onResult drop releases the single-flight gate so the queued re-convert proceeds.
- **F3 (LOW)** — `autoLevels()` had no error handling; a pica/getImageData throw was an unhandled rejection and the Auto button silently did nothing. Now surfaces via `host.showError`, generation/session-guarded like the convert catch.
- **F4 (COSMETIC)** — loupe label said "3.5× zoom"; actual magnification is 260/72 ≈ 3.6×. Label now derives from the constants; README/index.html updated.
- **F5 (cleanup)** — `displayScale.ts` private `getRotatedDims` replaced with the canonical `geometry.ts:rotatedSourceDims`.
