## Findings

### R4-1. `set-base-raster` failure permanently locks the convert pipeline ‚ÄĒ MEDIUM
Version-counter mismatch across the worker error path:
- `infra/worker/imageWorker.ts:22-28` ‚ÄĒ a `set-base-raster` failure posts an error tagged with the **sharedBufferVersion** (`msg.version`).
- `features/image/controller.ts:158` ‚ÄĒ `onError` drops any error whose version ‚Č† **processVersion**. The two counters are independent, so the error is (almost always) discarded as stale.
- `imageWorker.ts:34` ‚ÄĒ the `process` message that `convert()` posts right after returns **silently** when `baseRaster === null`: no result, no error.

Net effect: `processing` stays `true` forever; every later tweak only sets `processRequested`. Same failure class as N1 (round 1) and F2 (round 2), on the one remaining unguarded path. Low trigger probability (SAB view construction rarely throws) but total failure mode.

**Fix:** (1) worker: `process` with no base raster posts `{type:'error', phase:'process', version}` instead of returning silently; (2) controller `onError`: scope the processVersion staleness filter to `phase === 'process'`; validate `set-base-raster` errors against `runtime.sharedBufferVersion` instead (`-1` catch-all unchanged).

### R4-2. Mirror toggles on a 90¬į/270¬į-rotated image flip the wrong axis and lose the crop selection ‚ÄĒ MEDIUM
`buildRotatedSource` (`features/image/source.ts:27-31`) applies mirror **before** rotation (CTM = T¬∑R¬∑S ‚Üí mirror is source-space). At 90¬į/270¬į, a source-H flip is a **vertical** flip of the displayed view. But:
- The button says "Flip horizontal" ‚ÄĒ the user sees the perpendicular flip.
- `toggleMirrorH` (`controller.ts:204-210`) reflects the crop box **horizontally** in display coords while the content moved vertically ‚Üí the selection lands on different content. README ¬ßMirror explicitly promises preservation; verified broken by transform math (A/B pixel trace, 2√ó1 image at 90¬į: no-mirror A-top/B-bottom; mirrorH A-bottom/B-top).

At rotation 0/180 everything is consistent (at 180¬į, source-H flip ‚Č° display-H flip). No test pins mirror behavior today.

**Fix (view-space semantics ‚ÄĒ flip what you see):** when `rotation % 180 !== 0`, the H button dispatches `imageToggleMirrorV` and vice versa; each button keeps its display-axis box reflection (H ‚Üí `x' = dispW ‚ąí x ‚ąí w`, V ‚Üí `y' = dispH ‚ąí y ‚ąí h`), which then matches the actual content flip. Extract the reflection into pure `domain/geometry.ts:mirrorBoxRect` (sibling of `rotateBoxRect`) with unit tests. Update the index.html tooltip ("source" ‚Üí "view") and code-map ¬ß5.

State-consistency note: this keeps `mirrorH`/`mirrorV` as source-space flags (renderer unchanged); only the button‚Üíflag mapping rotates. Physical intuition holds: flipping a 90¬į-rotated photo horizontally, then rotating back, yields a vertically-flipped photo.

### R4-3. Histogram zones/markers use the hard-quantize triple even when dithering ‚ÄĒ MEDIUM-LOW
`controller.ts:147` and `bootstrap.ts:91` pass `getQuantThresholds(preset)` = `profile.thresholds`. Under `master` (the default since `3995f77`) with dithering enabled (also default), error diffusion actually bins at `ditherThresholds` `[30,50,140]`, not `[45,70,140]` ‚ÄĒ so the default configuration shows zone boundaries/markers that don't match what the dither does. (Under `pr1614` the triples coincide, which is why round 1's N2 fix looked complete.) Ordered modes (bayer/blue-noise/zf) interpolate against `ditherLevels`; their effective midpoints ‚Čą `[22.5, 55, 145]` ‚ÄĒ the dither triple is the closest single answer.

**Fix:** `domain/quantize.ts:getActiveQuantThresholds(preset, ditherEnabled)` ‚Üí `ditherEnabled ? profile.ditherThresholds : profile.thresholds`; use at both call sites (worker-result render + histogram auto-resize mount). Update code-map ¬ß6 quant entry.
