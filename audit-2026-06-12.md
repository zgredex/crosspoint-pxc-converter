# Audit — 2026-06-12 (round 4)

Scope: post-`3995f77` pass — the quant-profile/dithering commit and everything it touches (quantize, dither, worker protocol, histogram chain, preset toggle/badge), plus a fresh behavioral trace of the convert single-flight gate, mirror/rotation composition, crop interaction, GB pipeline, and app wiring. Baseline: `npm test` 122/122 green.

## Findings

### R4-1. `set-base-raster` failure permanently locks the convert pipeline — MEDIUM
Version-counter mismatch across the worker error path:
- `infra/worker/imageWorker.ts:22-28` — a `set-base-raster` failure posts an error tagged with the **sharedBufferVersion** (`msg.version`).
- `features/image/controller.ts:158` — `onError` drops any error whose version ≠ **processVersion**. The two counters are independent, so the error is (almost always) discarded as stale.
- `imageWorker.ts:34` — the `process` message that `convert()` posts right after returns **silently** when `baseRaster === null`: no result, no error.

Net effect: `processing` stays `true` forever; every later tweak only sets `processRequested`. Same failure class as N1 (round 1) and F2 (round 2), on the one remaining unguarded path. Low trigger probability (SAB view construction rarely throws) but total failure mode.

**Fix:** (1) worker: `process` with no base raster posts `{type:'error', phase:'process', version}` instead of returning silently; (2) controller `onError`: scope the processVersion staleness filter to `phase === 'process'`; validate `set-base-raster` errors against `runtime.sharedBufferVersion` instead (`-1` catch-all unchanged).

### R4-2. Mirror toggles on a 90°/270°-rotated image flip the wrong axis and lose the crop selection — MEDIUM
`buildRotatedSource` (`features/image/source.ts:27-31`) applies mirror **before** rotation (CTM = T·R·S → mirror is source-space). At 90°/270°, a source-H flip is a **vertical** flip of the displayed view. But:
- The button says "Flip horizontal" — the user sees the perpendicular flip.
- `toggleMirrorH` (`controller.ts:204-210`) reflects the crop box **horizontally** in display coords while the content moved vertically → the selection lands on different content. README §Mirror explicitly promises preservation; verified broken by transform math (A/B pixel trace, 2×1 image at 90°: no-mirror A-top/B-bottom; mirrorH A-bottom/B-top).

At rotation 0/180 everything is consistent (at 180°, source-H flip ≡ display-H flip). No test pins mirror behavior today.

**Fix (view-space semantics — flip what you see):** when `rotation % 180 !== 0`, the H button dispatches `imageToggleMirrorV` and vice versa; each button keeps its display-axis box reflection (H → `x' = dispW − x − w`, V → `y' = dispH − y − h`), which then matches the actual content flip. Extract the reflection into pure `domain/geometry.ts:mirrorBoxRect` (sibling of `rotateBoxRect`) with unit tests. Update the index.html tooltip ("source" → "view") and code-map §5.

State-consistency note: this keeps `mirrorH`/`mirrorV` as source-space flags (renderer unchanged); only the button→flag mapping rotates. Physical intuition holds: flipping a 90°-rotated photo horizontally, then rotating back, yields a vertically-flipped photo.

### R4-3. Histogram zones/markers use the hard-quantize triple even when dithering — MEDIUM-LOW
`controller.ts:147` and `bootstrap.ts:91` pass `getQuantThresholds(preset)` = `profile.thresholds`. Under `master` (the default since `3995f77`) with dithering enabled (also default), error diffusion actually bins at `ditherThresholds` `[30,50,140]`, not `[45,70,140]` — so the default configuration shows zone boundaries/markers that don't match what the dither does. (Under `pr1614` the triples coincide, which is why round 1's N2 fix looked complete.) Ordered modes (bayer/blue-noise/zf) interpolate against `ditherLevels`; their effective midpoints ≈ `[22.5, 55, 145]` — the dither triple is the closest single answer.

**Fix:** `domain/quantize.ts:getActiveQuantThresholds(preset, ditherEnabled)` → `ditherEnabled ? profile.ditherThresholds : profile.thresholds`; use at both call sites (worker-result render + histogram auto-resize mount). Update code-map §6 quant entry.

## Known limitations noted, no action
- Fit-size slider max derives from full-source dims; with a sub-source box selected in fit mode, the plan silently caps scale at 1 while the slider range overstates what's achievable. Display-only; fixing needs box dims threaded into `render` (state-side max field). Revisit if users notice.
- Quant-preset toggle while a GB file is loaded calls `requestConvert()` on the image controller — leaves a dangling `processRequested` that the next load resets. Harmless.

## Resolution log — 2026-06-12

All three findings fixed the same day (parallel implementation, disjoint file sets; verified by full suite + line-by-line diff review):

- **R4-1** — worker now posts a `phase:'process'` error (with the process version) when a `process` message arrives without a base raster, instead of returning silently; controller `onError` validates versions per phase (`process` → `processVersion`, `set-base-raster` → `sharedBufferVersion`, `-1` crash catch-all unchanged). Pinned by a controller test emitting both current and stale `set-base-raster` errors.
- **R4-2** — view-space mirror semantics: at 90°/270° the H button dispatches `imageToggleMirrorV` and vice versa; box reflection extracted to pure `domain/geometry.ts:mirrorBoxRect` (display-axis reflection always matching the button). index.html tooltip + README §Mirror updated; code-map §5 blessed-exception bullet documents the mapping. Pinned by geometry unit tests (reflection, involution, centered fixed point) and controller dispatch tests at rotation 0 and 90.
- **R4-3** — `domain/quantize.ts:getActiveQuantThresholds(preset, ditherEnabled)` returns the dither triple when dithering is on, the hard triple when off; used by both histogram render sites (worker `onResult`, `mountHistogramAutoResize`). code-map §3/§6 synced. Pinned by quantize unit tests over both presets × both dither states.
